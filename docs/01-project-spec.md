# 01 — 项目规格

> v2.2 | 2026-07-03 | 状态: ✅ 已完成

## 项目

企业流程工单管理系统，基于 RuoYi-Vue Spring Boot 3 二开。复用登录、权限、用户、部门、菜单等基础能力，新增 `ruoyi-ticket` 工单模块。

**解决的问题**：派活无记录难追溯、进度不透明、无法统计工单量、无统一服务入口。

## v1.0 功能范围

| 功能 | 说明 |
|---|---|
| 工单创建 | 标题、内容、分类、优先级 |
| 工单列表 | 分页 + 状态/优先级/分类/关键词/日期筛选 |
| 工单详情 | 基本信息 + 分类/用户/部门名称 + 评论 + 操作日志 |
| 工单分派 | 管理员指派处理人，NEW → PROCESSING |
| 工单处理 | 处理人提交结果，PROCESSING → WAIT_CONFIRM |
| 工单确认 | 创建人确认关闭，WAIT_CONFIRM → CLOSED |
| 工单取消 | 创建人或管理员取消，NEW/PROCESSING → CANCELLED |
| 评论 | 沟通记录，INTERNAL 内部备注 / EXTERNAL 公开评论 |
| 操作日志 | 每次流转记录 who / from / to / comment |
| 分类管理 | 树形分类，parentId + ancestors 路径 |

## v1.0 不做什么

SLA / 满意度 / 附件 / 模板 / 通知推送 / 批量操作 / 数据权限 / 父子工单 / ES 检索 / 导出 / Flowable / Python / RAG / Agent

## 工单状态流转

```
NEW ──分派──▶ PROCESSING ──处理──▶ WAIT_CONFIRM ──确认──▶ CLOSED
 │                 │
 └──取消───────────┘
```

- 5 个状态，5 条流转，`TicketStatus` 枚举内部 `allowedTransitions` 控制
- `CLOSED` / `CANCELLED` 终态
- 每次流转强制写 `ticket_operation_log`
- 不做可视化流程设计器，不做条件分支

## Redis 边界

v1.0 不引入新的 Redis 缓存。仅维持 RuoYi 原有：Token 持久化、验证码、在线用户统计。工单列表走 MySQL + PageHelper。

## 验收标准

1. 创建工单后 `ticketNo` 格式 `TK202606250001`，`status = NEW`
2. 分派后 `status = PROCESSING`，`assigneeId` 更新，`processedAt` 有值
3. 处理后 `status = WAIT_CONFIRM`，处理备注必填
4. 确认后 `status = CLOSED`，`closedAt` 有值
5. `NEW` / `PROCESSING` 可取消 → `CANCELLED`
6. 非法流转返回错误（终态不可操作、非指派人不可处理、非创建人不可确认/取消）
7. 每次流转 `ticket_operation_log` 一条记录
8. 详情返回完整信息（分类名、创建人/指派人昵称、部门名、评论、日志）
9. 列表支持多条件筛选和分页

| 功能 | 说明 |
|---|---|
| 流程模板 | 管理流程基本信息、节点、连线和发布版本 |
| 分类绑定 | 每个工单分类可绑定一个流程模板；未绑定时使用内置标准流程 |
| 版本快照 | 创建工单时锁定当时已发布版本，模板后续变更不影响存量实例 |
| 顺序节点 | 支持串行分派、处理和确认节点，不支持并行或会签 |

| 处理人规则 | 节点可指定用户、角色、创建人部门负责人、工单指派人或工单创建人 |
| 流程动作 | 支持完成当前节点、退回上一节点、取消和管理员终止 |
| 运行记录 | 保存流程实例、当前任务、处理人、处理意见和完整流转记录 |

## v2.0 规则

- 流程模板以 `workflow_key + version` 标识版本；草稿可编辑，发布后不可修改或删除。
- 同一流程模板同时只能有一个当前发布版本。发布新版本只影响之后创建的工单。
- 工单分类绑定稳定的 `workflow_key`，创建时解析当前发布版本并写入流程实例。
- 分类未绑定流程或绑定不可用时，使用系统内置标准流程；不得产生无流程工单。
- 条件仅读取服务端工单快照：`priority`、`category_id`、`dept_id`；不接受客户端 SQL 或表达式。
- 同一来源节点的条件按顺序匹配，最多命中一条；无条件命中时走唯一默认分支，否则事务回滚。
- 节点激活时解析处理人。用户、部门负责人、工单指派人和工单创建人固化为用户 ID；角色规则保留角色 ID，由具备该启用角色的用户处理。
- 业务状态继续使用 `NEW / PROCESSING / WAIT_CONFIRM / CLOSED / CANCELLED`：分派节点为 `NEW`，处理节点为 `PROCESSING`，确认节点为 `WAIT_CONFIRM`，到达结束节点为 `CLOSED`。
- 退回只允许退到当前实例已完成的直接上一人工节点；退回后重新生成任务，不回滚历史记录。
- 取消沿用创建人或管理员权限；终止仅管理员可执行，二者均结束流程并将工单置为 `CANCELLED`。
- v2.0 上线前的工单继续走原五状态流程，不创建流程实例，不迁移运行中数据。

## v2.0 不做什么

可视化拖拽设计器、并行网关、会签、或签、加签、转办、抄送、子流程、定时边界事件、任意脚本表达式、自定义字段条件、历史工单迁移、Flowable。

## v2.0 验收标准

1. 管理员可创建草稿、配置合法流程并发布；非法图、无默认分支或无处理人规则时拒绝发布。
2. 已发布版本不可修改；发布新版本后，旧工单仍按原版本继续运行。
3. 分类绑定流程后，新工单创建流程实例并生成唯一当前任务；未绑定分类使用标准流程。
4. 优先级、分类和创建人部门条件均能选择正确分支，且客户端不能注入条件表达式。
5. 五种处理人规则均正确鉴权，越权处理被拒绝。
6. 完成、退回、取消和终止在同一事务内更新任务、实例、工单状态及操作日志。
7. 重复提交同一任务不重复流转；并发处理时最多一个请求成功。
8. v1.0–v1.3 的 SLA、通知、评价、评论和数据权限回归通过。

| 功能 | 说明 |
|---|---|
| 字段定义 | 管理分类下的自定义字段、排序、必填、默认值和启停状态 |
| 字段类型 | 支持文本、数字、日期、日期时间、单选、多选和布尔值 |
| 选项管理 | 单选和多选字段维护稳定选项值、展示标签、排序和启停状态 |
| 创建校验 | 创建工单时校验未知字段、重复字段、必填、类型、长度、数值范围和选项 |
| 值快照 | 保存字段 key、名称、类型、值及选项标签快照，配置变更不追溯历史工单 |
| 详情展示 | 工单详情按字段排序返回自定义字段值和历史快照元数据 |
| 流程条件 | 动态流程支持按 `CUSTOM_FIELD` 的字段 key 和值执行 `EQ / IN` 路由 |

## v2.1 规则

- 自定义字段归属一个工单分类；`field_key` 在分类内唯一，格式为大写字母、数字和下划线。
- 字段创建后 `field_key`、`field_type` 和所属分类不可修改；名称、校验参数、排序和状态可修改。
- 字段不物理删除，只允许停用。停用字段不再出现在新工单表单中，历史值继续展示。
- 创建工单只接受服务端为该分类返回的启用字段；未知 key、重复 key 和越权分类字段一律拒绝。
- 文本仅支持最大长度；数字支持最小值和最大值；v2.1 不接收用户正则表达式。
- 单选和多选值保存稳定 `option_value`，展示时同时保存 `option_label` 快照。
- 多选值规范化为有序 JSON 数组；同一选项不得重复。
- 工单创建、字段值保存和流程实例启动必须处于同一事务，任一步失败全部回滚。
- 流程自定义字段条件缺失或类型不兼容时视为不命中，最终走唯一默认分支。

## v2.1 不做什么

字段公式、级联字段、动态正则、文件字段、富文本、字段级数据权限、列表动态列、按自定义字段筛选、批量导入、历史值回填。

## v2.1 验收标准

1. 管理员可为分类维护七种字段及选项，非法 key、重复 key 和非法校验参数被拒绝。
2. 创建工单时必填、类型、长度、数值范围、未知字段、重复字段和无效选项校验正确。
3. 工单、字段值和流程实例同事务保存，失败时不产生半成品数据。
4. 字段定义或选项修改后，历史工单仍按创建时快照展示。
5. 禁用字段不影响历史详情，且新工单不能提交该字段。
6. `CUSTOM_FIELD` 条件可正确执行文本、数字、选项和布尔值的 `EQ / IN` 路由。
7. 客户端不能通过字段 key、值或条件注入 SQL/脚本。
8. v1.0–v2.0 单元测试和完整 smoke 全部回归通过。

## v2.2 功能范围与规则

- 工单正文和评论均可关联附件；附件元数据保存在数据库，文件内容复用 RuoYi 本地文件存储。
- 保存原始文件名、相对存储路径、MIME、扩展名、大小和上传人快照，不向客户端暴露绝对路径。
- 临时上传按登录用户记录归属；列表、下载、绑定和删除均先执行工单对象级访问控制。
- 客户端不能直接提交服务器路径；临时附件只能由上传人绑定一次，且不得跨工单或跨业务类型绑定。
- 附件采用逻辑删除；终态工单附件默认不可删除，管理员除外。
- v2.2 不做分片上传、断点续传、秒传、在线预览、病毒扫描、对象存储或历史附件迁移。

## v2.2 验收标准

1. 有权用户可上传附件并绑定到工单正文或评论，非法类型、空文件和超限文件被拒绝。
2. 列表和下载遵守工单对象级权限，且下载响应不泄露服务器绝对路径。
3. 临时附件不能被他人、重复或跨工单绑定，业务保存失败时不产生错误绑定。
4. 原始文件名和文件元数据在绑定后保持快照，存储文件名不可由客户端控制。
5. 删除权限、终态限制和重复删除语义正确，物理文件清理失败可追踪。
6. v1.0–v2.1 单元测试和完整 smoke 全部回归通过。

## v1.3 规则

- 超级管理员可访问全部工单。
- 普通用户可访问“角色部门数据范围内的工单”与“本人创建或当前指派给本人的工单”的并集。
- “仅本人”同时包含本人创建和当前指派给本人的工单，保持 v1.0 的处理语义。
- 自定义部门继续使用 `sys_role_dept`，本部门及下级继续使用 `sys_dept.ancestors`，不新增数据范围配置表。
- 数据范围只认可启用且拥有对应 ticket 权限的角色；没有匹配角色时仅保留本人参与工单。
- 前端不得传入用户 ID、部门 ID 或原始 SQL 决定数据范围；服务端必须覆盖 `params.dataScope`。
- 对无权访问的工单统一按“工单不存在”处理，避免泄露工单是否存在。
- v1.3 不改变状态机、通知、SLA 和满意度规则，不新增数据库表或字段。

3. 跨部门指派后，当前指派人仍可查看和处理工单。
4. 普通用户不能通过查询参数、直接访问 ID 或关联接口读取范围外工单。

## v1.2 功能范围与规则

- v1.2 不接短信、邮件、企业微信；外部渠道需先确定供应商和密钥配置。

## v1.2 验收标准

1. 六类业务事件向正确用户生成通知，重复事件不重复通知。
2. 用户只能查询和标记自己的通知，未读数准确。
3. 创建人可对已关闭工单评价一次。
4. 非创建人、非关闭工单、重复评价和越界评分被拒绝。
5. 工单详情返回评价，管理员评价列表和统计正确。
6. v1.0、v1.1 smoke 全部继续通过。

## v1.1 功能范围

| 功能 | 说明 |
|---|---|
| SLA 策略 | 管理员按优先级配置首次响应时限、解决时限和启停状态 |
| 时限快照 | 创建工单时按优先级匹配策略，将响应和解决截止时间固化到工单 |
| SLA 展示 | 列表和详情展示截止时间、剩余时间及是否超时 |
| 超时扫描 | Quartz 每 5 分钟扫描一次活动工单，识别响应超时和解决超时 |
| 超时告警 | 每个工单、每种告警类型只生成一条站内告警，支持分页查询 |
| 手工补扫 | 管理员可手工触发一次扫描，便于运维和测试 |

## v1.1 规则

- SLA 按 `TicketPriority` 配置，`LOW / MEDIUM / HIGH / URGENT` 每个优先级最多一条策略。
- v1.1 统一按自然时间 24×7 计算，不处理工作日、节假日和暂停计时。
- 首次响应完成时间沿用 `processed_at`，即工单首次由 `NEW` 分派为 `PROCESSING` 的时间。
- 解决完成时间沿用 `closed_at`，即工单由 `WAIT_CONFIRM` 确认为 `CLOSED` 的时间。
- 响应截止时间和解决截止时间在创建工单时生成快照；之后修改策略不追溯已有工单。
- SLA 策略缺失或已停用时拒绝创建对应优先级的新工单，避免产生无时限工单。
- `CANCELLED` 工单退出 SLA 扫描；历史告警保留，不删除。
- 扫描任务必须幂等，唯一键保证同一工单同一告警类型只生成一次告警。
- 超时判断使用数据库时间：完成时间为空且当前时间晚于截止时间，或完成时间晚于截止时间。
- SLA 告警在 v1.1 只做站内记录和查询；外部消息渠道不在 v1.1/v1.2 范围。

## v1.1 不做什么

工作时间日历、节假日、暂停/恢复 SLA、多级升级、消息推送、告警确认、策略版本回溯、按分类或客户配置 SLA。

## v1.1 验收标准

1. 四种优先级均存在唯一、启用的默认 SLA 策略。
2. 创建工单时正确写入 `response_due_at` 和 `resolve_due_at`，且策略修改不改变已有工单。
3. 首次分派时间不晚于响应截止时间时，不产生响应超时告警。
4. 未分派且超过响应截止时间时，生成一条 `RESPONSE_OVERDUE` 告警并标记工单响应超时。
5. 未关闭且超过解决截止时间时，生成一条 `RESOLUTION_OVERDUE` 告警并标记工单解决超时。
6. 重复扫描不产生重复告警。
7. `CLOSED` 工单按实际关闭时间判断是否解决超时，`CANCELLED` 工单不再产生新告警。
8. 列表支持按响应超时、解决超时筛选，详情返回完整 SLA 信息。
9. 无权限用户不能维护策略、查看全部告警或手工触发扫描。
