# 01 — 项目规格

> v1.3 | 2026-07-02 | 状态: ✅ 已完成

## 项目

企业流程工单管理系统，基于 RuoYi-Vue Spring Boot 3 二开。复用登录、权限、用户、部门、菜单等基础能力，新增 `ruoyi-ticket` 工单模块。

**解决的问题**：派活无记录难追溯、进度不透明、无法统计工单量、无统一服务入口。

## v1.0 功能范围

| 功能 | 说明 |
|---|---|
| 工单创建 | 标题、内容、分类、优先级 |
| 工单列表 | 分页 + 状态/优先级/分类/关键词/日期筛选 |
| 工单详情 | 基本信息 + 分类/用户/部门名称 + 评论 + 操作日志 |
| 工单分派 | 管理员指派处理人，NEW → PROCESSING |
| 工单处理 | 处理人提交结果，PROCESSING → WAIT_CONFIRM |
| 工单确认 | 创建人确认关闭，WAIT_CONFIRM → CLOSED |
| 工单取消 | 创建人或管理员取消，NEW/PROCESSING → CANCELLED |
| 评论 | 沟通记录，INTERNAL 内部备注 / EXTERNAL 公开评论 |
| 操作日志 | 每次流转记录 who / from / to / comment |
| 分类管理 | 树形分类，parentId + ancestors 路径 |

## v1.0 不做什么

SLA / 满意度 / 附件 / 模板 / 通知推送 / 批量操作 / 数据权限 / 父子工单 / ES 检索 / 导出 / Flowable / Python / RAG / Agent

## 工单状态流转

```
NEW ──分派──▶ PROCESSING ──处理──▶ WAIT_CONFIRM ──确认──▶ CLOSED
 │                 │
 └──取消───────────┘
```

- 5 个状态，5 条流转，`TicketStatus` 枚举内部 `allowedTransitions` 控制
- `CLOSED` / `CANCELLED` 终态
- 每次流转强制写 `ticket_operation_log`
- 不做可视化流程设计器，不做条件分支

## Redis 边界

v1.0 不引入新的 Redis 缓存。仅维持 RuoYi 原有：Token 持久化、验证码、在线用户统计。工单列表走 MySQL + PageHelper。

## 验收标准

1. 创建工单后 `ticketNo` 格式 `TK202606250001`，`status = NEW`
2. 分派后 `status = PROCESSING`，`assigneeId` 更新，`processedAt` 有值
3. 处理后 `status = WAIT_CONFIRM`，处理备注必填
4. 确认后 `status = CLOSED`，`closedAt` 有值
5. `NEW` / `PROCESSING` 可取消 → `CANCELLED`
6. 非法流转返回错误（终态不可操作、非指派人不可处理、非创建人不可确认/取消）
7. 每次流转 `ticket_operation_log` 一条记录
8. 详情返回完整信息（分类名、创建人/指派人昵称、部门名、评论、日志）
9. 列表支持多条件筛选和分页

## v1.3 功能范围

| 功能 | 说明 |
|---|---|
| 角色数据范围 | 复用 RuoYi 角色的全部、自定义部门、本部门、本部门及下级、仅本人五种数据范围 |
| 工单列表隔离 | 列表仅返回角色数据范围内的工单，创建人和当前指派人始终可见其参与的工单 |
| 对象级鉴权 | 详情、评论、操作日志、评价详情等按工单 ID 查询的接口统一校验访问权 |
| 多角色合并 | 用户多个启用角色的数据范围取并集；任一角色拥有全部数据权限即不做部门过滤 |
| 部门快照 | 使用工单创建时写入的 `ticket.dept_id` 判定归属，用户后续调岗不追溯历史工单 |

## v1.3 规则

- 超级管理员可访问全部工单。
- 普通用户可访问“角色部门数据范围内的工单”与“本人创建或当前指派给本人的工单”的并集。
- “仅本人”同时包含本人创建和当前指派给本人的工单，保持 v1.0 的处理语义。
- 自定义部门继续使用 `sys_role_dept`，本部门及下级继续使用 `sys_dept.ancestors`，不新增数据范围配置表。
- 数据范围只认可启用且拥有对应 ticket 权限的角色；没有匹配角色时仅保留本人参与工单。
- 前端不得传入用户 ID、部门 ID 或原始 SQL 决定数据范围；服务端必须覆盖 `params.dataScope`。
- 对无权访问的工单统一按“工单不存在”处理，避免泄露工单是否存在。
- v1.3 不改变状态机、通知、SLA 和满意度规则，不新增数据库表或字段。

## v1.3 验收标准

1. 五种角色数据范围均按 RuoYi 现有配置生效。
2. 多角色范围正确取并集，停用角色和无 ticket 权限角色不扩大范围。
3. 跨部门指派后，当前指派人仍可查看和处理工单。
4. 普通用户不能通过查询参数、直接访问 ID 或关联接口读取范围外工单。
5. 用户调岗后，历史工单仍按创建时部门快照归属。
6. 管理员与普通用户的 v1.0、v1.1、v1.2 主流程回归通过。

## v1.2 功能范围与规则

- 站内定向通知：分派、处理完成、关闭、取消、评论、SLA 超时。
- 用户可查询自己的通知、未读数，并执行单条或全部已读。
- 通知不发给操作人自己，使用 `event_key + recipient_id` 保证幂等。
- 创建人可对 `CLOSED` 工单评价一次，评分为 1–5，可填写 500 字以内内容。
- 评价不可修改或删除；管理员可查询评价列表和满意度统计。
- v1.2 不接短信、邮件、企业微信；外部渠道需先确定供应商和密钥配置。

## v1.2 验收标准

1. 六类业务事件向正确用户生成通知，重复事件不重复通知。
2. 用户只能查询和标记自己的通知，未读数准确。
3. 创建人可对已关闭工单评价一次。
4. 非创建人、非关闭工单、重复评价和越界评分被拒绝。
5. 工单详情返回评价，管理员评价列表和统计正确。
6. v1.0、v1.1 smoke 全部继续通过。

## v1.1 功能范围

| 功能 | 说明 |
|---|---|
| SLA 策略 | 管理员按优先级配置首次响应时限、解决时限和启停状态 |
| 时限快照 | 创建工单时按优先级匹配策略，将响应和解决截止时间固化到工单 |
| SLA 展示 | 列表和详情展示截止时间、剩余时间及是否超时 |
| 超时扫描 | Quartz 每 5 分钟扫描一次活动工单，识别响应超时和解决超时 |
| 超时告警 | 每个工单、每种告警类型只生成一条站内告警，支持分页查询 |
| 手工补扫 | 管理员可手工触发一次扫描，便于运维和测试 |

## v1.1 规则

- SLA 按 `TicketPriority` 配置，`LOW / MEDIUM / HIGH / URGENT` 每个优先级最多一条策略。
- v1.1 统一按自然时间 24×7 计算，不处理工作日、节假日和暂停计时。
- 首次响应完成时间沿用 `processed_at`，即工单首次由 `NEW` 分派为 `PROCESSING` 的时间。
- 解决完成时间沿用 `closed_at`，即工单由 `WAIT_CONFIRM` 确认为 `CLOSED` 的时间。
- 响应截止时间和解决截止时间在创建工单时生成快照；之后修改策略不追溯已有工单。
- SLA 策略缺失或已停用时拒绝创建对应优先级的新工单，避免产生无时限工单。
- `CANCELLED` 工单退出 SLA 扫描；历史告警保留，不删除。
- 扫描任务必须幂等，唯一键保证同一工单同一告警类型只生成一次告警。
- 超时判断使用数据库时间：完成时间为空且当前时间晚于截止时间，或完成时间晚于截止时间。
- SLA 告警在 v1.1 只做站内记录和查询；外部消息渠道不在 v1.1/v1.2 范围。

## v1.1 不做什么

工作时间日历、节假日、暂停/恢复 SLA、多级升级、消息推送、告警确认、策略版本回溯、按分类或客户配置 SLA。

## v1.1 验收标准

1. 四种优先级均存在唯一、启用的默认 SLA 策略。
2. 创建工单时正确写入 `response_due_at` 和 `resolve_due_at`，且策略修改不改变已有工单。
3. 首次分派时间不晚于响应截止时间时，不产生响应超时告警。
4. 未分派且超过响应截止时间时，生成一条 `RESPONSE_OVERDUE` 告警并标记工单响应超时。
5. 未关闭且超过解决截止时间时，生成一条 `RESOLUTION_OVERDUE` 告警并标记工单解决超时。
6. 重复扫描不产生重复告警。
7. `CLOSED` 工单按实际关闭时间判断是否解决超时，`CANCELLED` 工单不再产生新告警。
8. 列表支持按响应超时、解决超时筛选，详情返回完整 SLA 信息。
9. 无权限用户不能维护策略、查看全部告警或手工触发扫描。
