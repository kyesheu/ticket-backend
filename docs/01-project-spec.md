# 01 — 项目规格与关键决策

> 版本: v1.0 | 日期: 2026-06-25 | 状态: 待实现

---

## 项目名称

企业流程工单管理系统

---

## 项目背景

很多企业内部客服、IT 支持、售后服务场景里，问题处理比较分散，可能通过微信群、飞书、电话或者 Excel 记录，容易出现漏单、重复处理、责任不清晰的问题。同时企业内部有大量知识文档和历史处理经验，但没有被很好地利用起来，新人处理问题时效率比较低。

所以 v1.0 先建设企业流程工单管理系统，基于 RuoYi 复用登录、权限、用户、部门、菜单、日志等基础能力，新增 ticket 工单模块，完成工单创建、分派、处理、确认、关闭/取消、评论和操作日志等主流程闭环。后续版本再接入知识库、RAG 和 Agent 辅助能力。

---

## 解决的问题

- 口头派活无记录、易遗漏、难追溯
- 请求方不知道进度，处理人不知道优先级
- 管理者无法统计工单量和处理效率
- 没有统一的服务入口和数据沉淀

---

## v1.0 目标

提供工单主流程闭环：创建 → 分派 → 处理 → 确认 → 关闭/取消。附带分类、评论、操作审计。

---

## v1.0 功能范围

| 功能 | 说明 |
|---|---|
| 工单创建 | 用户提交工单，填写标题、内容、分类、优先级 |
| 工单列表 | 分页查询，支持状态/优先级/分类/关键词/日期筛选 |
| 工单详情 | 查看工单完整信息、评论、操作历史 |
| 工单分派 | 管理员将工单指派给处理人 |
| 工单处理 | 处理人填写处理结果，提交待确认 |
| 工单确认 | 创建人确认处理结果，关闭工单 |
| 工单取消 | 创建人或管理员取消工单 |
| 工单评论 | 处理过程中添加沟通记录 |
| 操作日志 | 记录每次状态流转的 who / when / from / to / 备注 |
| 工单分类 | 树形分类管理 |

---

## 基于若依二开的原因

- RuoYi 提供成熟的 RBAC、JWT 认证、操作日志、接口文档基座，不重复造轮子
- Spring Boot 3 + Java 21 技术栈，与团队技术方向一致
- 模块化结构（Maven 多模块），新增业务模块不影响框架升级

---

## 复用若依的能力

| 能力 | 复用方式 |
|---|---|
| 用户 / 部门 / 岗位 | 直接使用 `sys_user` `sys_dept` `sys_post` |
| 角色与菜单权限 | `@PreAuthorize("@ss.hasPermi('...')")` 模式 |
| JWT 登录与 Token | Redis Token 持久化，不改动 |
| API 操作日志 | `@Log` 注解自动写入 `sys_oper_log` |
| 接口文档 | Springdoc + Swagger UI 自动扫描新 Controller |
| 分页 | PageHelper 自动分页 |

---

## 新增 ticket 模块做什么

在 `ruoyi-ticket` 模块中实现 4 张业务表 + 对应的 Controller / Service / Mapper：

- `ticket` — 工单主表
- `ticket_category` — 分类树
- `ticket_comment` — 评论
- `ticket_operation_log` — 业务操作审计日志

以新增 `ruoyi-ticket` 模块为主，必要时可对基础模块做小范围调整。

---

## v1.0 不做什么

- 不做 SLA 超时告警
- 不做满意度评价
- 不做批量分派 / 批量关闭
- 不做附件上传
- 不做工单模板
- 不做通知推送（短信 / 邮件 / 站内信）
- 不单独实现复杂数据权限规则
- 工单列表先按角色做基础范围控制：普通用户查看自己创建或指派给自己的工单，管理员查看全部工单；后续再接入若依 DataScope 做部门级数据权限
- 不做工单关联 / 父子工单
- 不做 ES 全文检索（列表搜索走 MySQL LIKE）
- 不做导出

---

## Redis 使用边界

v1.0 工单模块不引入新的 Redis 缓存。Redis 仅维持 RuoYi 原有能力：Token 持久化、验证码、在线用户统计。

工单列表查询直接走 MySQL + PageHelper 分页，数据量在万级别以内足够。

---

## 工单状态流转

5 个状态，5 条流转：

```
NEW ──分派──▶ PROCESSING ──完成处理──▶ WAIT_CONFIRM ──确认──▶ CLOSED
 │                 │
 └──取消───────────┘
```

- `CLOSED` 和 `CANCELLED` 为终态
- 流转规则用 Java 枚举内部控制（`allowedTransitions` 集合），Service 层统一校验
- 每次流转强制写入 `ticket_operation_log`

**不做可视化流程设计器，不做条件分支。** 5 条固定路径，代码足够。

---

## 不使用 Flowable 的原因

- v1.0 共 5 个状态、5 条硬编码流转，枚举即可维护
- Flowable 引入额外的数据表（ACT_* 70+ 张）、学习成本、运维复杂度
- 工作流引擎的价值在于动态流程、会签、并行分支——这些 v1.0 都不需要
- 后续若状态膨胀到 15+ 或需要自定义流程，再评估引入

---

## 不接入 Python / RAG / Agent 的原因

- v1.0 是标准的 CRUD + 状态机，无 AI/ML 需求
- 引入非 JVM 技术栈增加部署和运维复杂度
- 后续版本如有智能分派、自动分类等需求时再评估

---

## 操作日志为什么必须做

- 工单是跨角色的协作流程，每次流转需要可追溯
- RuoYi 的 `sys_oper_log` 记录的是 HTTP API 调用（运维视角），`ticket_operation_log` 记录的是业务动作（业务视角）
- 两者互补：出问题时运维查 API 日志，业务方查工单操作日志
- 审计合规：谁在什么时间把工单从什么状态改到了什么状态，必须可查

## v1.0 验收标准

1. 任意用户可创建工单，成功后工单状态为 `NEW`
2. 管理员可分派工单，状态变为 `PROCESSING`，指派人不可为空
3. 指派人可处理已分派的工单，状态变为 `WAIT_CONFIRM`，处理备注必填
4. 创建人可确认 `WAIT_CONFIRM` 状态工单，状态变为 `CLOSED`
5. `NEW` 或 `PROCESSING` 状态工单可被取消，变为 `CANCELLED`
6. 非法的状态流转返回错误，不可执行（如 CLOSED 后再处理、WAIT_CONFIRM 后再分派）
7. 非指派人不可处理工单，非创建人（且非管理员）不可确认/取消工单
8. 每次状态流转在 `ticket_operation_log` 中有一条记录
9. 工单详情接口返回完整信息：基本信息、分类、创建人、指派人、评论列表、操作日志
10. 工单列表支持按状态、优先级、分类、关键词筛选和分页
